## 场景规则

| 场景 | 首要来源 | 说明 |
|------|---------|------|
| APP UI（Edge Mobile 等） | `systems/mai.md` + 个人品味 | Design System 优先，个人品味填补空白 |
| 网页 / Landing Page | 个人品味优先 | 自由发挥，不受公司规范约束 |
| PPT / 汇报 | 信息层级优先 | 一页一个焦点，砍 30% 内容再砍 |
| 品牌 / 视觉传达 | 个人品味优先 | 最自由 |
| Dashboard / 数据可视化 | 清晰度优先 | 数字右对齐，文字左对齐，颜色表意 |

## Design Principles

七条原则。每个 UI 决策应该满足其中大部分。

### 1. 一个视图一个焦点
每个屏幕需要一个清晰的视觉锚点。如果所有东西都加粗，就没有重点。决定什么最重要，让它突出，其他一切为它服务。

### 2. 留白是结构，不是剩余
元素之间的空间传达关系。不是事后添加的 padding，而是有意设计的架构。塞满内容会摧毁信息层级。

### 3. 每个颜色必须有存在理由
如果你说不出为什么一个元素是这个颜色，它就不应该是。装饰性颜色是噪音，功能性颜色（状态、层级、强调、交互）是信号。

### 4. 能删掉且不影响的，就删掉
边框、阴影、图标、徽章、标签：质疑每一个元素。更少的元素做好，永远胜过更多元素互相竞争。

### 5. 先适应项目的现有设计系统
先读 codebase。如果有 token 系统、间距尺度、色板或组件库，扩展它，不要对抗它。没有系统时，默认精致极简 + 一个有个性的选择。

### 6. 不要产出模板化的东西
完成任何组件前问自己："这看起来属于 *这个* 项目，还是看起来像所有 AI 生成的应用？" 如果是后者，做一个刻意的改变赋予它个性。

### 7. 克制
默认从少开始。只在缺少会造成混乱时才添加元素。阴影 OR 边框，不要两个都加。一个动画、一个 hover 效果、一个强调色，通常就够了。

## Anti-Patterns

硬性禁止。如果发现自己在产出这些，停下来修改。

| Anti-Pattern | 是什么 | 应该怎么做 |
|---|---|---|
| **组件汤** | 一个 card 上堆 5-6 个元素互相竞争 | 一个标题 + 一个辅助信息 + 一个操作，从这里开始 |
| **模板既视感** | 千篇一律的 SaaS/Dashboard 外观 | 至少一个选择让它不能无缝换到其他应用 |
| **懒惰渐变** | 紫蓝渐变 hero、渐变文字、彩虹按钮 | 渐变用在大面积背景，柔和低饱和度，不用在按钮/文字/小元素上 |
| **过度装饰** | 阴影 + 边框 + 圆角 + 背景色 + hover 全上 | 每个元素选一个主要视觉处理。卡片用柔和阴影，不用 border |
| **神秘导航** | 图标没有文字标签 | 主导航用图标 + 文字。仅图标只用于公认符号（关闭、搜索、菜单） |
| **文字墙** | 长内容无分隔无层级 | 用标题分段、要点列表、提取关键数字 |
| **Transition: all** | `transition: all 0.3s ease` 到处用 | 只 transition 你要动的属性：`opacity`, `transform` |

## Pre-Output Checklist

出 UI 前的 7 个检查：

1. **层级** — 有没有一个清晰焦点？能不能追踪阅读顺序？
2. **颜色** — 每个颜色有功能目的吗？文字对比度 ≥ 4.5:1？
3. **间距** — 空间关系是有意的吗？相关项分组了吗？
4. **系统** — 遵循了项目现有 pattern 吗？所有值来自设计系统吗？
5. **克制** — 能删掉任何元素而不丢失意义吗？
6. **状态** — 考虑了 hover/focus/disabled/loading/empty/error 吗？
7. **响应式** — 在 375px、768px、1200px+ 上都能工作吗？

> 最后直觉检查："一个资深设计师会 ship 这个，还是会打回去再改一轮？"
